Year 3

Networking Turnbased Game

Task

In a group of 5 we had to use a game genre radomiser and then create a game based on that.

Weekly Reports

  • Sprint 1

    Reflecting from last game jam (Assassin Creed demake) we decided we need to organise the order of what coding gets done and when it needs to be done by so we can track out work. Trello does give this to a certain point but at my placement I used a piece of software called Jira which had all the functionality of Trello plus more. One key part was to be able to create sprints and add tasks into them and once the sprint was finish which was weekly for us, we could reflect on the work and decide what to do with the work that didn’t get completed.

    The next stage was to develop a game based on our random topic “A turn based strategy where you colonise ruins with cyborgs” and with this we decided to create a networked game where you take over ruins until the other player has none.

    We split this up into sections such as:

    • Networking
    • Map generation
    • Combat
    • Turn based system
    • Leader board
    • Challengers
    • Sound
    • UI
    • Sprites

    Then after we all got given our sections to code, we needed to create all the tasks needed to be done for the section to be completed. Then I created three sprints and we distributed them between the sprints making sure we all had enough work to do and that the work wasn’t too much to handle in three weeks. I was familiar with Jira, so I controlled the sprints and moved things around accordingly. On top of this we decided last game jam we didn’t meet up enough so every Monday before our Wednesday session we would meet and make sure everything was merged in to prevent cram coding and needing other people’s help when they are not free that evening.

    I was the most confident with networking, so I choose to do this because I have some experience from my placement with TCP and Socket connections as well as creating a message format for these.

    Scope was an important thing in this as creating a networking game is not quick so to keep the scope reasonable, we decided to have only two players with a Host/Client connection system.

    My first sprint was to have a client connecting to a host through TCP and to be able to send basic string messages to each other. I completed this before Wednesday and there was map generation in as-well, so we had the host generate the map randomly and then send the data over in XML to the client. The client would then build the exact map which put us in a good spot for the first week.

    One problem I found after a bit of testing is the map data would error occasionally because the “XML format was not complete”. Which means the message was not fully received before turning it back into XML format. The reason this was happening particular on the map message was because it was very long number of characters which meant it was split up into multiple packets. I wasn’t waiting for all packets received before turning back into XML.
